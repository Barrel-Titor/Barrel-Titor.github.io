用 Hexo 和 GitHub Page 建立简单的 Blog

发表于 2020-03-31 更新于 2020-05-13 分类于 不学无术 阅读次数： Valine：
本文字数： 4.2k 阅读时长 ≈ 4 分钟

每次折腾点新事物，都是疯狂踩坑的过程，但是也从中学到了不少。 用博客做点记录，方便以后同样踩坑的朋友解决问题，也方便自己日后回顾。


1 - 为什么选择 Hexo 和 GitHub Page

相信有心写博客的新手，一开始都会纠结选择哪个平台。就好像我打游戏建号的时候，纠结选用什么武器，最后总会发现，这 TM 根本不是重点。


1.1 - GitHub Page

对于多个平台的优缺点分析，有大佬已经做过了，有兴趣的朋友请自行移步。

我个人选用 GitHub Page 的理由：

  1. 不会有奇奇怪怪的审核君；
  2. 学习、熟悉一下 Web 开发和 GitHub 的使用
  3. 本身比较熟悉 Markdown 语法，撰写方便
  4. 免费QAQ


1.2 - Jekyll or Hexo?

在 GitHub Page 上有两个主流的静态博客框架：Jekyll 和 Hexo。Jekyll 与 GitHub 契合度更高，可以在 GitHub 中直接生成页面；Hexo 中文资料更丰富，基础功能更多，但是部署和备份比 Jekyll 麻烦。

Jekyll Hexo
页面生成 Github 直接生成静态页面 本地生成静态页面，然后上传
分支管理 一个分支即可 推荐使用两个分支，一个设置文件，一个页面文件

两者我都有浅尝，目前选用 Hexo，因为其自带了博客标签和分类的功能，而 Jekyll 需要自己设置 CSS 或者用插件实现。关于如何对 Hexo 进行版本控制，在后文有详细的讲解。

3 - 开始建站

GitHub Pages 存放的是静态文件，而我们的博客存放的不只是日志内容，还有列表、分类、标签等动态内容。每当我们撰写或修改日志，这些动态内容都应该更新。

Hexo 允许我们在本地撰写 Markdown 文件，写完后调用 Hexo 提供的命令，批量完成相关 HTML 页面的生成，最后将有改动的页面提交到 GitHub。
